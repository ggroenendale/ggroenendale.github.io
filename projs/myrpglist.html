    My RPG List

    Summary

    The idea of this project is to create a platform where Table Top Role Players can share the stories they are telling with their favorite game systems and fans can find other stories their favorite players are connected to.


    Stakeholders:

    • Fans
    • Players and Game Masters
    • Game Creators and Content Writers

    Problem

    • Content Creators and Players are often at the mercy of content algorithms on video platforms and social media for drawing fans to their content.
    • Players often play in one campaign without viewers knowing where else they can find other appearances.
    • Game Designers, Module Writers, Artists also have to fight the algorithms for getting their creations into the spotlight. There are many more game systems than D&D and Pathfinder.
    • Viewers have a hard time finding characters, genres, and stories they most associate with. Western themes, characters with similar gender identities and sexuality, certain species or races and character classes.

    Requirements

    1. The entire system is secure
      1. SSL Certificates
      2. Secret Keys
      3. Secure Passwords for Services
    2. Users are able to log in to the platform:
      1. Normal users with favorite content
      2. Content Owner users who are able to edit content they are associated with
      3. Administrative users who have elevated privileges to delete content
    3. A frontend view for each type of content - Campaign, Character, Game System, Actor, etc.
    4. Content can be searched and results displayed
    5. File Storage for images and a Content Delivery Network (CDN) that caches these files geographically near users
    6. A Data Storage solution that makes content relationships easily queryable
    7. Data Validation built into the API for creating and retrieving records
    8. Data Pipeline for
      1. Updating new episodes
      2. Finding new campaigns and channels
      3. Finding game systems
      4. Finding new campaign settings and adventure modules
      5. Finding newly created character classes and species
    9. Content and information can be created for different languages as well as searchable by language.
      1. This means localization and internationalization to display the same text in different languages.
      2. Also content should be searchable by what language is spoken in the episode so users can find it.

    System Design Overview

    The below diagram details the basic system design behind this platform.


    On the far left are the user inputs. The three types of users satisfies the user requirements in Requirement 2. The diagram also depicts 2 types of front end platforms for Users and Admins.


    • The User Front End WebApp allows normal users to view various types of content and link to their sources. This app also allows content creators and content managers to show and hide content as well as edit metadata. To manage traffic demand the front end app will be containerized and behind a load balancer.
    • The Admin Front End allows administrators to view all records and assign content managers as well as edit metadata. Additionally, this web app includes analytics and viewable logs.

    The Front End WebApps are written in Python using the Flask library for routes and view templates written with Jinja to insert data onto the webpages. In order for a view to have different data for records of the same content type, the webapp and api utilize text slugs instead of id's to request and retrieve data. Using text slugs in the urls for content pages makes them more seo friendly. This satisfies Requirement 3.


    In order to satisfy the search feature in Requirement 4, ElasticSearch is used for text searches and returns a text slug or uuid in order to link to the specific view.


    The Content Delivery Network caches images and frequent web pages to make the user experience smoother. This satisfies Requirement 5.


    The data storage solution depends on a relational database to find relationships with foreign keys. The model is built using the SQLAlchemy library as the ORM. This has additional features to make some relationships easy to manage. Additionally, the Alembic library makes it easy to handle additions and changes to the database. This satisfies Requirement 6.


    Data Validation is handled using the Pydantic library which comes with FastAPI. This will validate data when creating and updating records and requesting records.


    There are a few options for a data pipeline and this project will use a combination of Python scripts and Apache Airflow. Workflows and tasks in Airflow are written in Python, but additional scripts may be developed outside of Airflow to retrieve data.

    Data Model and Storage

    This schema details the relationships between the SQL tables in the PostgresQL database. The ORM model is built using SQLAlchemy which adds some functionality to make the relationships easier to manage. For example, content managers are handled using a mixin relationship table. So instead of creating a special model for each content and user association, a class is implemented for each content which creates an intersection table for each content type that will need a content manager.

    Progress

    Progress - Aug 13, 2023
    Requirement 1 - Not Started
    While in development SSL Certificates and server security has not been necessary. A plan for security by server and by service is in development. A well documented data security plan for ElasticSearch and PostgresQL will be the most critical.
    Requirement 2
    Requirement 2.1 - Initial Development
    A front end platform for regular users is in the early stages of design and development.
    Requirement 2.2 - Complete
    Additionally, the ability to assign content ownership of content to a user is complete.
    Requirement 2.2 - Usable
    The admin front end is much further along than the normal front end. Data can be edited and managed using the admin front end. Logs are also viewable, while pipeline information and management needs further development.
    Requirement 3 - Initial Development
    As the user front end is in early development, views have not been completed. There's also an expectation that as the data pipeline is further developed some data will be added to the model to eventually be displayed on the front end.
    Requirement 4 - Complete
    This is mostly implemented. A search route on the backend API has been created to consume search terms and sent to elastic for results. There are indexes for each type of content in elastic and the indexes are periodically updated on a time basis from the backend. The schema for each index could be modified to enable more search results for the same term. For example making sure an actor name is in a campaign record so when searching for the actor a campaign result is displayed as well as the actor result. But the feature of the search route returning a slug to then be linked to a view route works.
    Requirement 5 - Not Started
    While developing the basic system and data model images have taken a low priority. A CDN won't be implemented until a basic system is deployed and working for users.
    Requirement 6 - Operational
    The data model and storage solution has been thoroughly developed for each content type. Indexes and relationship properties are built into the model. A plan to backup and restore data needs to be finalized and documented. Currently, the admin front end web application stores flat files for each content type for data backup. Data restore needs to be finalized especially in cases where data entities have several foreign keys.
    Requirement 7 - Operational
    Data validation is also thoroughly handled with specific Pydantic models for creating and updating records. Pytest tests need to be completed for each type of content to ensure things likes special characters and realistic date limits.
    Requirement 8 - Initial Development
    The data pipeline is one of the next features to be developed. At this time some proof of concept functions have been written into the admin front end but as some external service api's do charge for access or usage this has had a lower priority than the back end system and front end design.
    Requirement 9 - Not Started
    Localization and Internationalization has also taken a low priority as the back end and front end systems need to be completed.
